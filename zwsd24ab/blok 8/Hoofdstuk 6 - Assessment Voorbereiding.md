# Hoofdstuk 6

## Assessment Voorbereiding

Aan het einde van Blok 8 sluit je af met een **mondeling assessment**. Dit hoofdstuk helpt je te begrijpen wat er van je verwacht wordt en hoe je je goed kunt voorbereiden.

---

## Wat is het mondeling assessment?

Het is een gesprek van ca. 15–20 minuten tussen jou en je docent. Het gaat over:

1. **Jouw bijdrage** — wat heb jij persoonlijk gebouwd?
2. **Technische kennis** — begrijp je de code die geschreven is, ook het deel van je teamgenoten?
3. **Scrum-proces** — hoe heb je de Scrum-methode toegepast?
4. **Reflectie** — wat ging goed, wat zou je een volgende keer anders doen?

Je wordt **individueel** beoordeeld. Dat je in een team hebt gewerkt is fijn, maar in het assessment ben je op jezelf.

---

## Mogelijke vragen

### Over jouw bijdrage
- Welke user stories heb jij gebouwd?
- Welke technische uitdaging tegengekomen en hoe heb je die opgelost?
- Laat me de code zien van [specifieke functie] — leg uit hoe het werkt.

### Over de applicatie
- Hoe werkt de authenticatie in jullie applicatie?
- Hoe zijn de gebruikersrollen geregeld?
- Wat doet [model] en welke relaties heeft het?
- Als ik als admin dit wil doen, welke route/controller/view wordt dan aangeroepen?

### Over Scrum
- Wat is de rol van een Scrum Master en wie was dat in jouw team?
- Wat stond er in jullie Sprint Backlog?
- Wat is het verschil tussen een Sprint Review en een Sprint Retrospective?
- Welke verbeterpunten hebben jullie na Sprint 1 afgesproken? Zijn die uitgekomen?

### Over reflectie
- Wat vond je het moeilijkste in dit blok?
- Wat zou je een volgende keer anders aanpakken in de samenwerking?
- Wat heb je geleerd over werken in een team?

---

## Hoe bereid je je voor?

### 1. Ken je eigen code
Loop door alle code die jij hebt geschreven. Kun je het uitleggen? Kun je uitleggen waarom je bepaalde keuzes hebt gemaakt?

Oefen deze zin: *"Ik heb gekozen voor [aanpak] omdat [reden]."*

### 2. Begrijp de code van je team
Je hoeft niet elk detail te weten, maar je moet de grote lijn begrijpen. Vraag teamgenoten om uitleg als iets niet duidelijk is. Lees de commit history op GitHub.

### 3. Bereid een korte demo voor
Weet hoe je de applicatie doorloopt als een normale gebruiker. Wat zijn de drie belangrijkste functies die je kunt laten zien?

### 4. Kijk terug op het Scrum-proces
Lees de Sprint Backlogs terug. Wat heb je gepland, wat is er af, wat niet? Kun je dat verklaren?

### 5. Wees eerlijk
Als iets niet werkt, zeg dat. Als je iets niet begrijpt, zeg dat. Een eerlijk antwoord ("dat weet ik niet precies, maar ik denk dat het zo werkt...") is beter dan iets verzinnen.

---

## Checklist voor de laatste sprint

Gebruik deze checklist in de laatste projectweek:

**Applicatie**
- [ ] De applicatie start zonder fouten
- [ ] Login en registratie werken
- [ ] Alle user stories van Sprint 1 én Sprint 2 zijn afgerond of bewust uitgesteld
- [ ] Er zijn geen broken links of lege pagina's
- [ ] De code staat op GitHub (main branch is up-to-date)

**Scrum**
- [ ] Product Backlog is actueel in de scrumtool
- [ ] Sprint Backlog van Sprint 2 is volledig bijgewerkt
- [ ] Retrospective van Sprint 1 is gedocumenteerd

**Demo**
- [ ] Je weet welke functionaliteit je wilt laten zien
- [ ] Je hebt de demo minstens één keer geoefend
- [ ] Er staat testdata in de database zodat de demo er goed uitziet

**Persoonlijke voorbereiding**
- [ ] Je kunt per user story die je hebt gebouwd uitleggen hoe de code werkt
- [ ] Je kunt de database-structuur uitleggen (tabellen, relaties)
- [ ] Je kunt uitleggen hoe jullie de samenwerking hebben geregeld

---

## Opdracht 1 – Persoonlijke samenvatting

Schrijf voor jezelf (niet inleveren, wel nuttig) een korte samenvatting:

1. Welke user stories heb ik gebouwd?
2. Wat was de moeilijkste technische uitdaging die ik tegenkwam?
3. Wat heb ik geleerd over samenwerken in een team?
4. Wat zou ik in een volgend project anders doen?

---

## Opdracht 2 – Code walkthrough

Kies de drie functies/pagina's in de applicatie die jij hebt gebouwd en die je het meest trots op bent. Leg aan een teamgenoot uit hoe de code werkt: van route → controller → model → view. Als jij het kunt uitleggen aan iemand anders, kun je het ook uitleggen in het assessment.

---

## Succes!

Je hebt in dit blok voor het eerst als een echt development team gewerkt. Dat is niet altijd makkelijk — samenwerken is een vak apart. Het feit dat je het gedaan hebt, is al een prestatie. Laat zien wat je hebt gebouwd en vertel eerlijk hoe het ging.
