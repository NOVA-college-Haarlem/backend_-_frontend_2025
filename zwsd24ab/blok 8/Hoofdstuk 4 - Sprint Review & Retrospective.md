# Hoofdstuk 4

## Sprint Review & Retrospective

Dit hoofdstuk hoort bij het einde van **Projectweek 1** en het begin van de tweede lesperiode. Je hebt drie dagen gebouwd. Nu is het tijd om terug te kijken — op het product én op het proces.

---

## Sprint Review

### Wat is het?

De Sprint Review is een demo van wat je team heeft gebouwd. Je laat de werkende software zien aan de Product Owner (je docent). Geen PowerPoint, geen uitleg van wat je *van plan* was — je laat zien wat er **nu** staat en wat het doet.

De PO geeft feedback:
- Wat is goed opgeleverd?
- Wat voldoet nog niet aan de verwachting?
- Wat gaat er in de volgende sprint in?

### Hoe bereid je het voor?

Een Sprint Review duurt ca. 15–20 minuten per groep. Bereid het volgende voor:

1. **Overzicht van de sprint**: welke user stories waren gepland, welke zijn afgerond?
2. **Demo**: loop door de applicatie alsof je een nieuwe gebruiker bent. Laat de happy flow zien.
3. **Obstakels**: wat is niet afgerond en waarom?
4. **Vragen voor de PO**: is er iets onduidelijk, of wil je feedback op een specifieke keuze?

> **Tip**: Oefen de demo vooraf. Niets is zo onhandig als live zoeken waar de knop zit.

### Wat telt als "klaar"?

Een user story is klaar als:
- De acceptatiecriteria zijn afgevinkt
- De functie werkt in de browser (geen errors, geen half-gebouwde schermen)
- De code staat op GitHub (gepushed, gemerged naar main)

Halverwege gebouwde features tellen **niet** mee. Dat is Scrum. Liever minder, maar afgerond.

## Sprint Retrospective

### Wat is het?

Na de Review kijk je met het team terug op hoe het samenwerken ging — niet wat je hebt gebouwd, maar hoe. Dit is een eerlijk gesprek, geen afrekening.

Het doel: je leert van de afgelopen sprint zodat de volgende beter gaat.

### Start / Stop / Continue

Gebruik deze eenvoudige structuur:

| | Vraag | Voorbeelden |
|--|-------|-------------|
| **Start** | Wat moeten we gaan doen wat we nog niet deden? | Elke dag een stand-up houden, eerder om hulp vragen, feature branches gebruiken |
| **Stop** | Wat moeten we stoppen met doen? | Wachten tot de dag voor de deadline, allemaal in dezelfde file werken |
| **Continue** | Wat werkte goed en houden we vast? | Duidelijke taakverdeling, goede communicatie via WhatsApp, ERD gemaakt vóór het bouwen |

### Hoe voer je een retro uit?

1. Iedereen schrijft stil zijn punten op (3–5 minuten) — dit zorgt dat iedereen bijdraagt, ook de stillere teamleden.
2. Iedereen deelt zijn punten, je verzamelt ze per categorie.
3. Kies samen **maximaal 3 verbeterpunten** voor de volgende sprint. Meer dan dat kun je toch niet onthouden.
4. Schrijf de afspraken op in de scrumtool of in een gezamenlijk document.

> **Valkuil**: Een retro waar iedereen zegt "het ging prima" leert je niets. Durf eerlijk te zijn — ook over jezelf.

---

## Opdracht 1 – Sprint Review voorbereiden

Bereid als team de Sprint Review voor:

1. Maak een overzicht: welke user stories waren gepland, welke zijn volledig afgerond?
2. Bepaal de volgorde van de demo: welke functionaliteit laat je als eerste zien?
3. Wie doet de demo? (niet iedereen tegelijk — kies één persoon die navigeert)
4. Wie presenteert de obstakels?

---

## Opdracht 2 – Retrospective uitvoeren

Voer de retro uit met je team:

1. Iedereen schrijft 5 minuten stil zijn Start / Stop / Continue punten op.
2. Bespreek de punten. Luister zonder te onderbreken.
3. Kies samen de **drie belangrijkste verbeterpunten**.
4. Schrijf op hoe jullie deze verbeteringen gaan doorvoeren in Sprint 2.

Lever de uitkomst van de retro in bij je docent (een korte notitie is genoeg).

---

## Opdracht 3 – Sprint 2 Planning

Na de Review en Retro start je direct met de planning van Sprint 2.

1. Bekijk samen met de PO welke user stories er nog openstaan en welke er nieuw bijkomen.
2. Update de Product Backlog in de scrumtool.
3. Kies welke stories je in Sprint 2 oppakt.
4. Splits ze op in taken en verdeel het werk.

**Let op**: verwerk ook de feedback van de Sprint Review. Zijn er dingen die de PO wil aanpassen of toevoegen? Dat gaat bovenaan de backlog.
